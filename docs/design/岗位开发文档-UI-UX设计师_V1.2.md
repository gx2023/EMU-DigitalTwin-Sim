## 1. 岗位职责概述

UI/UX设计师在本项目中承担以下核心职责：

### 1.1 系统整体UI/UX设计

- 负责动车组检修数字孪生仿真系统的整体视觉风格定义和用户体验架构设计
- 制定统一的设计规范体系（Design System），确保全系统界面风格一致
- 建立和维护设计组件库，提升设计效率和开发一致性
- 定义系统的信息架构和导航体系，确保用户能够高效完成任务

### 1.2 各功能模块的界面设计

- 段厂仿真模块：3D场景管理界面、调车仿真界面、台位管理界面、数据看板界面
- 检修作业模块：工艺流程编辑器、工艺卡片展示、作业管控界面、AI识别结果展示
- 部件检修仿真界面：拆装仿真界面、检测数据展示界面、方案对比界面
- 设备运行模块：设备监控界面、预警管理界面、维护管理界面
- 平台基础功能：登录界面、权限管理界面、系统配置界面

### 1.3 3D场景交互设计

- 设计3D场景中的交互逻辑和操作流程
- 定义3D场景中的信息展示方式（标注、气泡、面板等）
- 设计3D场景与2D界面的融合交互方案
- 制定3D场景的视角切换、聚焦、漫游等交互规范

### 1.4 大屏可视化设计

- 设计段厂运营数据大屏的整体布局和视觉风格
- 设计各类数据可视化图表（折线图、柱状图、雷达图、热力图、甘特图等）
- 设计大屏端的信息层次和数据展示优先级
- 确保大屏在3-8米视距下的可读性和视觉冲击力

### 1.5 设计规范制定与维护

- 编写和维护UI/UX设计规范文档
- 建立设计评审机制，确保设计质量
- 与开发团队协作，确保设计稿的准确还原
- 持续收集用户反馈，迭代优化设计方案

## 2. 用户画像与场景

> 以下用户画像与场景完整提取自PRD文档第2章，UI/UX设计师需基于这些画像进行针对性的界面设计。

### 2.1 段厂管理人员

#### 2.1.1 用户画像

| 属性 | 描述 |
|------|------|
| **角色名称** | 段厂管理人员（车间主任/副主任） |
| **年龄范围** | 35-50岁 |
| **技术背景** | 具备10年以上铁路运维经验，熟悉段厂运营管理 |
| **使用频率** | 每日使用，每次30分钟-2小时 |
| **使用终端** | PC端（办公室）、大屏端（调度中心） |
| **核心诉求** | 实时掌握段厂运营状态，优化资源调度，提升检修效率 |
| **痛点** | 信息分散、调度依赖经验、缺乏全局视角 |

**UI/UX设计要点：**
- 需要提供清晰的全局运营视图，信息层次分明
- 大屏端设计需突出关键运营指标，支持远距离快速识别
- 操作流程需简洁高效，减少不必要的交互步骤
- 数据看板需支持自定义，满足不同管理场景需求

#### 2.1.2 典型场景

**场景一：日常运营监控**

> 王主任每天早上8:00到达调度中心，通过大屏查看段厂数字孪生系统。系统实时展示当前所有台位的占用情况、在修动车组信息、检修进度、人员分布等。王主任可以快速了解段厂整体运营状态，发现异常情况及时处理。

**设计要求：**
- 大屏首页需一屏展示段厂运营全景
- 核心KPI指标需醒目展示，支持实时刷新
- 异常状态需使用醒目颜色标识（黄色预警、红色故障）
- 3D场景需支持实时状态动态展示

**场景二：检修计划调度**

> 根据当日动车组运用计划和检修需求，王主任在系统中查看各台位的预计空闲时间，通过仿真推演功能模拟不同的台位分配方案，选择最优方案后下达调度指令。

**设计要求：**
- 台位占用状态需以甘特图或时间轴形式直观展示
- 仿真推演结果需支持多方案对比展示
- 调度指令下达流程需简洁明确
- 需提供台位空闲时间的可视化预测

**场景三：资源调配决策**

> 当出现突发情况（如某台位设备故障）时，王主任通过系统快速查看其他台位的可用情况，通过仿真分析评估对检修计划的影响，制定应急调整方案。

**设计要求：**
- 突发情况需有明显的预警提示
- 备选方案需以对比表格形式展示
- 影响评估结果需直观可视化
- 应急调整操作需有确认机制

**场景四：运营数据分析**

> 每周例会上，王主任通过系统生成的运营分析报告，向团队展示本周检修完成情况、台位利用率、检修质量指标等数据，分析存在的问题并制定改进措施。

**设计要求：**
- 运营报告需支持多维度数据展示
- 图表需支持同比环比分析
- 报告需支持导出（PDF/Excel格式）
- 数据需支持钻取查看详情

### 2.2 检修工程师

#### 2.2.1 用户画像

| 属性 | 描述 |
|------|------|
| **角色名称** | 检修工程师（技术员/工艺工程师） |
| **年龄范围** | 28-45岁 |
| **技术背景** | 具备5年以上动车组检修经验，熟悉检修工艺和技术标准 |
| **使用频率** | 每日使用，每次1-4小时 |
| **使用终端** | PC端（办公室/现场） |
| **核心诉求** | 优化检修工艺，管理工艺资料，指导现场作业 |
| **痛点** | 工艺资料管理分散，工艺验证缺乏有效手段 |

**UI/UX设计要点：**
- 工艺流程编辑器需支持可视化拖拽操作，降低使用门槛
- 工艺卡片需结构化展示，信息层次清晰
- 仿真验证结果需直观展示，支持快速定位问题
- 数据分析图表需专业且易读

#### 2.2.2 典型场景

**场景一：工艺流程建模**

> 李工程师负责编制新的二级修工艺流程。他在系统中使用工艺流程建模工具，通过拖拽方式定义检修工序、工步和作业要求，系统自动生成标准化的工艺卡片。完成后，他通过仿真功能验证工艺流程的合理性，检查是否存在工时冲突或资源瓶颈。

**设计要求：**
- 工艺流程编辑器需提供丰富的节点类型和连接方式
- 拖拽操作需流畅，支持撤销/重做
- 工艺流程需支持自动布局和手动调整
- 仿真验证结果需以可视化方式展示冲突和瓶颈

**场景二：工艺卡片管理**

> 李工程师在系统中维护结构化工艺卡片库。每张工艺卡片包含作业名称、工序步骤、所需工具物料、质量标准、注意事项等结构化信息。当检修规程更新时，他可以快速定位受影响的工艺卡片并进行批量修改。

**设计要求：**
- 工艺卡片需采用卡片式布局，信息分区明确
- 支持按车型、修程、工序等多维度筛选和搜索
- 批量操作需提供确认和预览机制
- 版本变更需有清晰的对比展示

**场景三：检修作业仿真验证**

> 在引入新的检修工艺前，李工程师在数字孪生环境中进行仿真验证。他设置虚拟动车组和检修环境，按照新工艺流程进行仿真推演，评估作业时间、资源需求和潜在风险，验证工艺可行性。

**设计要求：**
- 仿真参数配置界面需直观易用
- 仿真过程需支持实时进度展示
- 仿真结果需以多维度报告形式展示
- 需支持仿真过程的加速/减速/暂停控制

**场景四：检修数据分析**

> 李工程师定期通过系统的统计分析功能，分析各工序的作业时间、质量合格率、返工率等数据，识别工艺薄弱环节，提出改进建议。

**设计要求：**
- 统计图表需支持多维度切换和下钻
- 薄弱环节需以醒目方式标识
- 趋势分析需支持时间范围选择
- 分析结果需支持导出和分享

### 2.3 检修工人

#### 2.3.1 用户画像

| 属性 | 描述 |
|------|------|
| **角色名称** | 检修工人（检修作业人员） |
| **年龄范围** | 22-45岁 |
| **技术背景** | 具备基本的动车组检修操作技能，正在持续提升 |
| **使用频率** | 每日使用（VR培训不定期），每次10-30分钟 |
| **使用终端** | VR设备（培训）、AR设备/Mobile（现场辅助） |
| **核心诉求** | 获得清晰的作业指导，提升操作技能 |
| **痛点** | 作业指导不够直观，特殊工况缺乏练习机会 |

**UI/UX设计要点：**
- VR界面需注重沉浸感和操作引导的清晰性
- AR界面需确保信息叠加不影响现场作业安全
- 操作指引需简洁明了，步骤化展示
- 需考虑现场环境光线、噪音等对交互的影响

#### 2.3.2 典型场景

**场景一：VR检修培训**

> 新员工小张在VR培训室进行转向架检修培训。他戴上VR头显，进入虚拟的检修车间，按照系统引导完成转向架一系弹簧检查作业。系统实时提示操作要点，记录操作过程，培训结束后给出评分和改进建议。

**设计要求：**
- VR培训界面需提供清晰的操作引导箭头和文字提示
- 当前操作步骤需高亮显示，已完成步骤置灰
- 操作反馈需及时（视觉+声音）
- 培训评分界面需直观展示得分和改进建议

**场景二：AR检修辅助**

> 检修工人赵师傅在现场进行制动盘检查作业时，使用AR眼镜查看叠加在实车上的数字化工艺指导信息，包括检查点位标注、标准间隙数值、历史检查记录等。AR辅助帮助他更准确地完成检查作业。

**设计要求：**
- AR标注需精准对齐实物，偏差需控制在可接受范围内
- 信息叠加需简洁，避免遮挡关键作业区域
- 需支持语音或手势切换显示内容
- 历史数据叠加需以趋势图形式展示

**场景三：作业过程记录**

> 检修工人完成每个工步后，在系统中确认完成并记录作业结果。对于关键工序，系统自动采集作业数据（如扭矩值、间隙测量值等），实现作业过程的数字化记录。

**设计要求：**
- 工步确认操作需简单快捷（一键确认）
- 数据录入界面需支持自动填充和校验
- 关键数据需有合格/不合格的醒目标识
- 电子签认流程需简洁安全

**场景四：故障处理练习**

> 在VR培训环境中，小张练习处理转向架异响故障。系统模拟故障现象，小张需要按照故障处理流程进行排查和修复。系统记录他的排查路径和修复操作，评估其故障处理能力。

**设计要求：**
- 故障现象模拟需逼真（视觉+听觉）
- 排查路径需提供分支选择和提示
- 操作评估需覆盖排查逻辑和操作规范性
- 需提供标准排查路径的对比展示

### 2.4 设备维护人员

#### 2.4.1 用户画像

| 属性 | 描述 |
|------|------|
| **角色名称** | 设备维护人员（设备工程师/维修工） |
| **年龄范围** | 25-50岁 |
| **技术背景** | 具备机电设备维修经验，熟悉段厂各类检修设备 |
| **使用频率** | 每日使用，每次20分钟-1小时 |
| **使用终端** | PC端（办公室）、Mobile（现场巡检） |
| **核心诉求** | 实时掌握设备运行状态，及时发现和处理设备故障 |
| **痛点** | 设备故障发现滞后，维护计划缺乏数据支撑 |

**UI/UX设计要点：**
- 设备状态需以颜色编码方式直观展示
- 预警信息需醒目且分级展示
- 设备详情需支持快速切换和对比
- 移动端界面需适配现场巡检场景

#### 2.4.2 典型场景

**场景一：设备状态监控**

> 设备工程师陈工每天通过系统查看段厂所有检修设备的运行状态。系统以3D模型形式展示各设备的位置和状态，绿色表示正常运行，黄色表示预警，红色表示故障。陈工可以点击设备查看详细的运行参数。

**设计要求：**
- 3D场景中设备状态需使用标准颜色编码（绿/黄/红）
- 设备列表需支持按状态、类型、区域快速筛选
- 设备详情面板需展示关键运行参数和趋势图
- 需支持从3D场景一键跳转到设备详情

**场景二：设备故障预警**

> 系统检测到不落轮镟床主轴振动值持续升高，触发预警。陈工收到预警通知后，在系统中查看设备运行趋势图，分析振动值变化规律，判断可能存在的故障原因，提前安排检修。

**设计要求：**
- 预警通知需醒目弹出，包含预警等级和设备信息
- 趋势图需支持多参数叠加对比
- 预警详情需提供可能的故障原因分析
- 需支持一键创建维护工单

**场景三：设备维护计划管理**

> 陈工在系统中编制月度设备维护计划。系统根据设备运行数据和维护历史，自动推荐需要维护的设备和建议的维护时间。陈工确认后，系统自动生成维护工单并通知相关人员。

**设计要求：**
- 维护计划需以日历/甘特图形式展示
- 系统推荐需以醒目方式标识，支持一键采纳
- 维护工单创建流程需简洁高效
- 需支持维护计划的拖拽调整

**场景四：设备能耗分析**

> 陈工通过系统的能耗分析功能，查看各设备的能耗数据和趋势。发现某台空压机能耗异常偏高，通过分析运行记录，发现存在空转时间过长的问题，建议优化运行策略。

**设计要求：**
- 能耗数据需以图表形式直观展示

#### 3.5.1 大屏规格

#### 3.5.2 大屏设计原则

| 原则 | 说明 |
|------|------|
| 信息层次清晰 | 重要信息突出展示，次要信息适当弱化 |
| 数据可视化为主 | 以图表、数字、3D视图为主要展示形式 |
| 减少文字描述 | 尽量用图形化方式传达信息 |

## 8. 设计验收标准

### 8.1 设计规范验收标准

| 验收项 | 验收标准 | 验证方法 |
|--------|---------|---------|
| 设计规范完整性 | 覆盖色彩、字体、间距、组件、图标、布局、交互等全部规范 | 文档审查 |
| 设计规范一致性 | 全系统界面风格统一，符合设计规范 | 设计走查 |
| 组件库完整性 | 基础组件、业务组件覆盖所有功能模块 | 组件库审查 |
| 图标库完整性 | 图标覆盖所有功能模块，风格统一 | 图标库审查 |

### 8.2 页面设计稿验收标准

| 验收项 | 验收标准 | 验证方法 |
|--------|---------|---------|
| 设计稿覆盖率 | 所有功能页面均有对应设计稿 | 需求对照 |
| 设计稿保真度 | 高保真设计稿，包含所有交互状态 | 设计评审 |
| 响应式适配 | 支持不同分辨率适配 | 多分辨率测试 |
| 设计稿规范性 | 符合设计规范，命名规范、图层规范 | 设计走查 |

### 8.3 交互设计验收标准

| 验收项 | 验收标准 | 验证方法 |
|--------|---------|---------|
| 交互流程完整性 | 核心业务流程均有交互原型 | 原型演示 |
| 操作效率 | 常用操作不超过3步完成 | 任务测试 |
| 反馈及时性 | 操作结果在1秒内给予反馈 | 交互测试 |
| 错误容忍 | 支持操作撤销，关键操作二次确认 | 交互测试 |
| 3D交互流畅性 | 3D场景交互帧率≥30FPS（全景）/≥60FPS（局部） | 性能测试 |
| 键盘快捷键 | 所有定义的快捷键功能正常 | 功能测试 |

### 8.4 大屏设计验收标准

| 验收项 | 验收标准 | 验证方法 |
|--------|---------|---------|
| 大屏布局合理性 | 信息层次清晰，核心信息突出 | 专家评审 |
| 远距离可读性 | 3-8米视距下所有信息可读 | 实际测试 |
| 数据可视化准确性 | 图表数据准确，更新及时 | 数据验证 |
| 大屏性能 | 4K分辨率下流畅运行，数据刷新≥1次/秒 | 性能测试 |
| 色彩对比度 | 深色背景下前景色对比度充足 | 色彩测试 |

### 8.5 VR/AR设计验收标准

| 验收项 | 验收标准 | 验证方法 |
|--------|---------|---------|
| VR界面可读性 | VR中文字清晰可读，面板布局合理 | VR设备实测 |
| VR交互流畅性 | VR场景帧率≥90FPS | VR设备实测 |
| VR舒适度 | 无明显晕眩感，移动和旋转舒适 | 用户测试 |
| AR标注精度 | AR标注与实物对齐偏差在可接受范围内 | 现场测试 |
| AR信息清晰度 | AR叠加信息清晰可读，不影响作业 | 现场测试 |

### 8.6 设计交付物验收标准

| 验收项 | 验收标准 | 验证方法 |
|--------|---------|---------|
| 设计稿完整性 | PC端≥60页、大屏端≥10页、VR端≥15页、AR端≥8页 | 数量统计 |
| 图标库完整性 | ≥200个系统图标、≥50个设备图标 | 数量统计 |
| 切图资源完整性 | PC端+大屏端全部切图资源 | 资源审查 |
| 交互原型完整性 | 核心流程均有交互原型 | 原型审查 |
| 设计文档完整性 | 设计规范、交互说明、3D场景设计说明等文档完整 | 文档审查 |

### 8.7 可用性验收标准

| 指标 | 要求 | 验证方法 |
|------|------|---------|
| 新用户上手时间 | ≤2小时（基础功能） | 用户测试 |
| 培训周期 | ≤3天（完整功能培训） | 培训记录 |
| 用户满意度 | ≥85分（满分100分） | 问卷调查 |
| 操作错误率 | ≤5%（标准作业流程） | 任务测试 |
| 界面一致性 | 全系统界面风格统一 | 设计走查 |
| 无障碍设计 | 符合WCAG 2.1 AA级标准 | 无障碍测试 |
