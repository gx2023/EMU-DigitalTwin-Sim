## 1. 岗位职责概述

后端开发工程师（.NET方向）在本项目中承担以下核心职责：

### 1.1 微服务后端架构设计与开发

- 负责基于 .NET 8 / ASP.NET Core 的微服务后端架构设计与实现
- 负责各微服务模块的拆分、独立部署及服务间通信机制设计
- 负责API网关的设计与配置，统一管理接口路由、鉴权、限流等
- 负责消息队列（RabbitMQ / Kafka）的集成，实现服务间异步通信与削峰填谷

### 1.2 数据库设计与数据管理

- 负责系统数据库的表结构设计、索引优化及数据模型维护
- 负责关系数据库（PostgreSQL / 达梦 / 人大金仓）的开发与兼容适配
- 负责时序数据库（TimescaleDB / TDengine）的设备运行数据存储方案设计
- 负责Redis缓存策略设计与实现
- 负责数据库分库分表策略及历史数据归档方案

### 1.3 API接口设计与开发

- 负责RESTful API接口的设计、开发与文档编写（Swagger / OpenAPI）
- 负责gRPC服务接口的定义与实现（微服务间通信）
- 负责WebSocket实时推送接口的开发
- 负责API接口的版本管理、限流、熔断等治理策略

### 1.4 与外部系统的数据集成

- 负责与EMIS系统、TEDS系统、PHM系统的API对接与数据同步
- 负责与SCADA系统、PLC控制器的工业协议（OPC UA / MQTT）数据采集对接
- 负责数据导入导出功能的开发（Excel/CSV/PDF等格式）
- 负责第三方系统集成的标准API接口开发

### 1.5 IoT数据采集与处理

- 负责MQTT Broker（EMQX）的部署与集成
- 负责IoT传感器数据的接收、解析、存储与转发
- 负责设备实时数据的高并发写入优化
- 负责数据采集链路的监控与异常处理

### 3.2 数据模型

#### 3.2.1 核心实体关系描述

**实体关系总览：**

```
段厂（depot_info）
  ├── 1:N ── 股道（track_info）
  ├── 1:N ── 检修台位（workstation_info）
  ├── 1:N ── 检修设备（equipment_info）
  ├── 1:N ── 动车组（emu_info）
  └── 1:N ── 组织架构（organization）

动车组（emu_info）
  ├── 1:N ── 检修工单（maintenance_order）
  └── 1:N ── 检修档案（maintenance_archive）

检修工单（maintenance_order）
  ├── N:1 ── 动车组（emu_info）
  ├── N:1 ── 检修台位（workstation_info）
  ├── 1:N ── 检修记录（maintenance_record）
  ├── 1:N ── AI识别记录（ai_detection_record）
  └── 1:N ── 工单附件（order_attachment）

工艺流程（process_definition）
  ├── 1:N ── 工序定义（step_definition）
  └── 1:N ── 工艺卡片（process_card）

工序定义（step_definition）
  ├── N:1 ── 工艺流程（process_definition）
  ├── N:N ── 前置工序（自关联）
  └── 1:N ── 检修记录（maintenance_record）

检修设备（equipment_info）
  ├── N:1 ── 段厂（depot_info）
  ├── 1:N ── 设备运行数据（equipment_runtime_data）
  ├── 1:N ── 设备维护记录（equipment_maintenance_record）
  └── 1:N ── 设备故障记录（equipment_fault_record）

用户（sys_user）
  ├── N:1 ── 角色（sys_role）
  ├── N:1 ── 组织架构（organization）
  ├── 1:N ── 检修记录（maintenance_record）
  └── 1:N ── VR培训记录（vr_training_record）

培训课程（training_course）
  ├── 1:N ── 培训场景（training_scene）
  └── 1:N ── VR培训记录（vr_training_record）
```


#### 5.2.2 编码规范

- 遵循微软官方 C# 编码规范
- 使用 C# 12 最新语法特性（主构造函数、集合表达式等）
- 控制器仅负责HTTP请求/响应处理，业务逻辑放在Service层
- 使用DTO进行数据传输，禁止直接暴露实体类
- 所有公共API方法必须编写XML注释
- 使用FluentValidation进行请求参数校验
- 使用AutoMapper进行对象映射
- 异步方法统一使用 async/await 模式
- 使用Polly实现重试、熔断、超时等弹性策略

#### 5.2.3 数据库开发规范

- 使用EF Core Code First进行数据库版本管理（Migration）
- 所有表必须有主键、创建时间、更新时间字段
- 软删除字段统一命名为 `is_deleted`，类型为 `bool`
- 外键字段命名规则：`{关联表主键名}`
- 索引命名规则：`idx_{表名}_{字段名}`
- 数据库操作必须使用事务，确保数据一致性
- 复杂查询使用存储过程或视图优化性能
- 禁止在循环中执行数据库查询（N+1问题）

#### 5.2.4 API开发规范

- RESTful API URL使用小写字母和连字符（kebab-case）
- API版本管理通过URL路径实现：`/api/v1/{resource}`
- 统一响应格式：`{ code, message, data, timestamp }`
- 统一错误响应格式：`{ code, message, errors, timestamp }`
- HTTP状态码规范使用：200（成功）、201（创建）、400（参数错误）、401（未认证）、403（无权限）、404（不存在）、500（服务器错误）
- 分页参数统一：`page`（页码，从1开始）、`pageSize`（每页条数）
- 排序参数统一：`sortBy`（排序字段）、`sortOrder`（asc/desc）

## 6. 接口设计规范

### 6.1 RESTful API设计规范

#### 6.1.1 URL设计

### 6.2 WebSocket实时推送规范

#### 6.2.1 连接管理

- WebSocket连接地址：`wss://{host}/ws/{hub}`
- 连接认证：通过Query参数传递JWT Token：`?token={jwt_token}`
- 心跳机制：客户端每30秒发送ping，服务端返回pong
- 断线重连：客户端实现指数退避重连策略

### 6.3 gRPC服务间通信规范

#### 6.3.1 Proto文件规范

```protobuf
syntax = "proto3";

package emudtsim.workstation;

service WorkstationService {
  rpc GetWorkstation (GetWorkstationRequest) returns (WorkstationResponse);
  rpc ListWorkstations (ListWorkstationsRequest) returns (WorkstationListResponse);
```

#### 6.3.2 服务治理

- 所有gRPC服务必须实现健康检查协议（GRPC Health Checking）
- 服务发现使用Consul / Kubernetes Service
- 调用超时设置：默认5秒，长耗时操作可配置为30秒
- 使用gRPC拦截器实现日志记录、链路追踪、异常处理

### 6.4 数据交换格式

#### 6.4.1 JSON格式规范

- 使用标准JSON格式，字符编码UTF-8
- 日期时间格式：ISO 8601（`2026-04-20T10:30:00Z`）
- 数值类型：整数使用int，浮点数使用decimal，避免精度丢失
- 枚举值：使用字符串形式（如 `"status": "occupied"`）
- 空值处理：null值字段可以省略或使用null

#### 6.4.2 Protobuf格式规范

- 微服务间通信优先使用Protobuf格式
- Protobuf消息定义遵循Google API Design Guide
- 版本兼容性：新增字段使用optional，不删除旧字段
